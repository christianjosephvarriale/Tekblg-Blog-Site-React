The N x N Queens Problem and IPC

  • 2020-02-16T11:50+00:00
  • By Tekblg

Let's have some fun with algorithms

We're going to talk about a classic computer science problem which is also deeply rooted in theoretical math, or discrete math as taken in CS / Eng classrooms in Canada. We're going to touch upon the fundamentals behind backtracking, shared memory, semaphores, syntax and debugging using GDB

Algorithm Background

The N x N queens problem goes as follows. How many configurations can you place N queens on an N x N chess board such that none of the queens can attack each other. For those of you who aren't smart and have never played chess, I'm kidding, a queen can attack any other piece on a chess board if it's along the same diagonal, or if it's on the same row or column. In school we break it down into the logical propositions. We're not going to do that here because we're not in school nor do we like theoretical math

Backtracking

Let's first get the english down, translate that to logical propositions for illustration purposes, then get started on the other aspects of this project. This algorithm is more generally known as a SAT solver to a Boolean satisfiability problem, which implements backtracking. Backtracking is recursive in nature, although you CS kids would know any recursive algorithm can be converted into an iterative one, Firstly, you place a queen at Q11 on the board, and check to see if safe( ), since there are no queens on it's diagonal, or column, or row then safe( ) returns true. Now we save the current state of the board, and iterate through the next available spaces with the next available N - 1 queens. In the more general version we would continue attempting to place queens on that same row until we've reached the end, however there are different flavours to this algorithm to increase it's efficiency, SAT solvers are O( 2 ^ n ), so in the interest of the health and safety of our motherboard we will assume the current row is no longer valid because two queens can't exist on the same row. We would then move to the second row, first column, and find that the queen can't be placed there. We will then move onto the next column, check that, and so on. If we come to a place where the board is no longer solvable with the current configuration, we backtrack by removing the previous queens position and attempt to replace it on the board. Here's the algorithm in c code.

int safe(char * config, int i, int j)
{
    int r, s;
    for (r = 0; r < i; r++)
    {
        s = config[r];
        if (j == s || i-r==j-s || i-r==s-j)
            return 0;
    }
    return 1;
}
int count = 0;
void nqueens(char *config, int n, int i)
{
    char *new_config;
    int j;
    if (i==n)
    {
        count++;
    }
    
    /* try each possible position for queen <i> */
    for (j=0; j<n; j++)
    {
        /* allocate a temporary array and copy the config into it */
        new_config = malloc((i+1)*sizeof(char));
        memcpy(new_config, config, i*sizeof(char));
        if (safe(new_config, i, j))
        {
            new_config[i] = j;
	        nqueens(new_config, n, i+1);
        }
        free(new_config);
    }
    return;
} 
int main(int argc, char *argv[])
{
    int n;
    char *config;
    if (argc < 2)
    {
        printf("%s: number of queens required\n", argv[0]);
        return 1;
    }
    n = atoi(argv[1]);
    config = malloc(n * sizeof(char));
    printf("running queens %d\n", n);
    nqueens(config, n, 0);
    printf("# solutions: %d\n", count);
    return 0;
}

IPC

So that's great, we've got a fancy algorithm which solves our problem. This can be made interesting however. What if instead of running this entire algorithm in a single process, we ran it in two separate processes? That way, since it's a poor exponential runtime, we can speed up the process and hopefully not run out of resources during execution. Let's try that.

In c, in order to create two different processes there's a function called fork( ). The function will create two different processes and copy the current local variables values, not references, into each process as they each have their own local variables, stack frame, etc. If we give the safe( ) method to one process, let's call it the child, and the nqueens( ) method to the other, let's call it the parent, then the problem is solved.. right?

Wrong! Once the child has finished executing how will the parent receive it's return value? They can't share variables in the traditional way because they each have their own copy. What's worse, even if they could share variables how do you ensure the child computes safe( ) when required? How would you ensure the parent knows when it can safely check it's 'return' value? The answer is shared memory and semaphores.

Shared Memory & Semaphores

Shared memory is memory which is, well.. shared! It's shared between processes and the OS must be let known ahead of time of this for proper allocation and deallocation. Since fork gives each process a copy of the previous variables, we'd have access to a shared memory pointer if it was declared before the fork. We also need a way to store the different levels of recursion, converting it to a sort of iterative algorithm because we need to declare memory and therefore size ahead of time. If we utilize some sort of structure which can contain both a 'global' state and a 'recursive' config state if you will, and organize the config state based on the stack depth then voila! you've got yourself code. Let's see what that might look like

Here's the declaration of the relevant structures :

struct shm_struct {
    int csl; /* current stack level in is_safe */
    int is_done; /* when parent hangs up */
    int is_safe; /* when the config is safe */
    int count;
    sem_t cld_barrier;
    sem_t par_barrier;
    struct config * config;
};
struct config
{
    int i;
    int j;
    char* data;
};

Here you can see we maintain an array of config objects which hold the recursive local variables for the nqueens( ) function call. We also have a couple of flags to indicate when we've exhausted all possibilities for configurations, and a 'global' way for the safe( ) method to indicate whether the given config based on config->i and config->j is safe through int is_safe; Now we're going to declare our shared memory, wrap it in the shm_struct and therefore both processes can manipulate those variables.

How do we ensure synchronization then? How does the parent process know when int is_safe; is accurate? The answer is semaphores. Obviously there are many ways to do this and this is by no means an exhaustive guide to advanced synchronization patterns but it does give a common use case to a common configuration known as a producer-consumer configuration. The parent process requests for is_safe( ) to be calculated, then waits for it's signal, or post. The child process waits until it's requested by the parent, updates the is_safe( ) marker, then waits for the next wakeup. The semaphore library let's you do just that. It allows a process to block, wait, if the value of a semaphore is 0 and post in order to wake-up a process waiting on that semaphore. If our initial state is to have a parr and a chd sem which control the processes, then we can make this work. The full algorithm is shown below.

Full Algorithm

// from the Cilk manual: http://supertech.csail.mit.edu/cilk/manual-5.4.6.pdf
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/shm.h>
#include <sys/wait.h>
#include <semaphore.h>
#include <pthread.h>
#include <unistd.h>
struct shm_struct {
    int csl; /* current stack level in is_safe */
    int is_done; /* when parent hangs up */
    int is_safe; /* when the config is safe */
    int count;
    sem_t cld_barrier;
    sem_t par_barrier;
    struct config * config;
};
struct config
{
    int i;
    int j;
    char* data;
};

void safe(struct shm_struct * container)
{
    int i = container->config[container->csl].i;
    int j = container->config[container->csl].j;
    int r, s;
    for (r = 0; r < i; r++)
    {
        s = container->config[container->csl].data[r];
        if (j == s || i-r==j-s || i-r==s-j) {
            container->is_safe = 0;
            return;
        }
    }
    container->is_safe = 1;
}
void nqueens(struct shm_struct * container, int n, int i)
{
    int j;
    int csl;
    // increment the call stack counter
    container->csl++;
    csl = container->csl;
    container->config[csl].i = i;
    if (i==n)
    {
        container->count++;
    }
    /* try each possible position for queen <i> */
    for (j=0; j<n; j++)
    {
        /* allocate a temporary array and copy the config into it */
        container->config[csl].j = j;
        char * old_data = container->config[csl-1].data;
        char * new_data = container->config[csl].data;
        memcpy(new_data, old_data, i);
        // now we wait for is_safe to return
        sem_post(&container->cld_barrier);
        sem_wait(&container->par_barrier);
        if (container->is_safe) {
            container->config[csl].data[i] = j;
	        nqueens(container, n, i+1);
        }
        // clear the shared memory;
        memset(container->config[csl].data, 0, (n + 1) * sizeof(char));
    }
    container->csl--; 
    if (container->csl  == 0) {
        container->is_done = 1;
        sem_post(&container->cld_barrier);
    }
    return;
}
int main(int argc, char *argv[])
{
    int n;
    key_t key;
    if (argc < 2)
    {
        printf("%s: number of queens required\n", argv[0]);
        return 1;
    }
    n = atoi(argv[1]);
    int shm_config_container_id = shmget(IPC_PRIVATE, sizeof(struct shm_struct), IPC_CREAT | 0666);
    
    // the maximum number of config structs is equal to n + 2 because the first is empty
    int shm_config_id = shmget(IPC_PRIVATE, sizeof(struct config) * (n + 2), IPC_CREAT | 0666);  
    struct shm_struct * container = shmat(shm_config_container_id, NULL, 0);
    container->config = shmat(shm_config_id, NULL, 0);
    // the largest size of the tracking array is n + null char 
    int shm_data_id[n + 1];
    for (int t = 0; t < (n + 2); t++) {
        shm_data_id[t] = shmget(IPC_PRIVATE, sizeof(char) * (n + 2), IPC_CREAT | 0666);
        container->config[t].data = shmat(shm_data_id[t], NULL, 0);
        container->config[t].i = 0;
        container->config[t].j = 0;
    }
    // configure container
    sem_init(&(container->cld_barrier), 1, 0);
    sem_init(&(container->par_barrier), 1, 0);
    container->is_done = 0;
    container->is_safe = 0;
    container->csl = 0;
    container->count = 0;
    
    int pid = fork();
    if (pid == 0){ /* Child Process */
        while (!(container->is_done)){
            sem_wait(&(container->cld_barrier));
            safe(container);
            sem_post(&(container->par_barrier));
        }
        for (int t = 0; t < (n + 2); t++) {
            shmdt(container->config[t].data);
        }
        shmdt(container->config);
        shmdt(container);
    } else { /* Parent Process */
        printf("running queens %d\n", n);
        nqueens(container, n, 0);
        printf("# solutions: %d\n", container->count);
        
        for (int t = 0; t < (n + 2); t++) {
            shmdt(container->config[t].data);
            shmctl(shm_data_id[t], IPC_RMID, NULL);
            
        }
        shmdt(container->config);
        shmdt(container);
        shmctl(shm_config_container_id, IPC_RMID, NULL);
        shmctl(shm_config_id, IPC_RMID, NULL);
    }
    return 0;
}
