React Router and Rails Server

  • 2020-04-20T00:00-04:00
  • By tekblg

Let's build a SEO friendly site

React Router is a popular library for creating pages in react. By default, react employs a single page application structure, where an application rendered by ReactDOM.render() is rendered to a root element on the index.html file. It's important to determine whether that's optimal for the type of site you're trying to build. From a UX perspective, single page applications allow for a more centralized user experience where they can access your site's content quickly. It also increases the probably of any individual piece of content to be seen, as users are less likely to see content the more clicks it takes to reach it

Single Page Apps Pros

When are single page apps good? Interactive applications like facebook, instagram, and gmail are good examples of single page apps. Here, users want large amounts of data to be readily accessible. Think about how engaging your facebook news feed would be if you had to put in effort through clicks to find cat videos... ya no. Another excellent use case is a landing page for a new product. Here, we may want to highlight our products features, and want a potential investor to have all the facts outlined to them quickly before we lose their engagement. Overall, not a bad strategy.

Single Page Apps Cons

If you're facebook or gmail or a well funded startup with lots of publicity, then traffic might not be your chief concern. People are going to find you with or without search engine's boost. However if you're not a well known site, like a blog website trying to create a name for yourself, then you don't have this luxury. There are a couple of issues with single page applications:

  1. Google can't decide what the page is about if there's tons of content, thereby being difficult to include in a relevant search query

  2. You miss out on an SEO benefit where the URL help page ranking

  3. You can't implement static site generation

All of these points are related to SEO, where search engines are a huge potential traffic source for your site. Think about it, social media can only take you so far. If I create a viral blog post, maybe I get tons of engagement. Maybe that engagement lasts for a week, but after that, it may dissipate to close to 0 recurring viewers. That's not sustainable, and neither is relying on the consistent efforts of constant social media backlinking. Some research suggests that many small-medium user base sites get up to 75% of their traffic from organic sources such as Google. That's big

Philosophy

React Router is how we can create URL's like https://tekblg.com/blog/gallery/technology/computerengineering/1 as opposed to just all our content being displayed on https://tekblg.com. React Router has well documented API's and a handful of easy to configure components to make the routing process be as simple as possible. Here are the docs: https://reacttraining.com/react-router/web/guides/quick-start

Common Components

export default function BasicExample() {
  return (
    <Router>
      <div>
        <ul>
          <li>
            <Link to="/">Home</Link>
          </li>
          <li>
            <Link to="/about">About</Link>
          </li>
          <li>
            <Link to="/dashboard">Dashboard</Link>
          </li>
        </ul>
        <hr />
        {/*
          A <Switch> looks through all its children <Route>
          elements and renders the first one whose path
          matches the current URL. Use a <Switch> any time
          you have multiple routes, but you want only one
          of them to render at a time
        */}
        <Switch>
          <Route exact path="/">
            <Home />
          </Route>
          <Route path="/about">
            <About />
          </Route>
          <Route path="/dashboard">
            <Dashboard />
          </Route>
        </Switch>
      </div>
    </Router>
  );
}
// You can think of these components as "pages"
// in your app.
function Home() {
  return (
    <div>
      <h2>Home</h2>
    </div>
  );
}
function About() {
  return (
    <div>
      <h2>About</h2>
    </div>
  );
}
function Dashboard() {
  return (
    <div>
      <h2>Dashboard</h2>
    </div>
  );
}

A couple of components to show off here, but this is the basic way you'll use React Router. Here we have a BasicExample() function which will return JSX containing several Link components. Link is what the page get's changed to onClick, and you can think about it as an <a href="path/to/url">link</a> for internal pages within your react app.

Whenever a <Link /> or <NavLink /> component is used, it must be wrapped in a <BrowserRouter /> or <HashRouter /> component as is shown above with the more general <Router /> component. Typically, you'd wrap your <App /> component in those tags like so, then directly render that in your index.js file like so

import React from 'react';
import ReactDOM from 'react-dom';
import './index.css';
import App from './App';
import * as serviceWorker from './serviceWorker';
import { BrowserRouter } from 'react-router-dom';

ReactDOM.render(    
<BrowserRouter
  basename={optionalString}
  forceRefresh={optionalBool}
  getUserConfirmation={optionalFunc}
  keyLength={optionalNumber}>
  <App />
</BrowserRouter>, document.getElementById('root'));

Now there must be a way for React to know which component to render based on the URL, this is accomplished by using the <Switch /> statement. Think of it like a switch statement in general. If one of the cases evaluates to true, then render that page. We use a series of <Route> component's, and if the path is matched to the path caught by the <BrowserRouter> then that one is rendered. Note two key props on <Route> both exact and path. path determines essentially the case for matching in the switch statement, which can be variable if the path contains url params.

For example. Say that we'd like https://tekblg.com/blog/gallery/technology/software and https://tekblg.com/blog/gallery/technology/hardware rendered by the <Blog /> component. In this case, we need both those URL's to match this component, and we can treat it as containing a url-param

export default function URLParams() {
  return (
    <Router>
        <Switch>
          <Route exact path="/">
            <Home />
          </Route>
          <Route path="/blog/gallery/technology/:catagory">
            <Blog />
          </Route>
          <Route path="/another/url">
            <OtherComponent />
          </Route>
        </Switch>
      </div>
    </Router>
  );
}

Note if the first Route didn't have the exact prop, then all routes would match that route. Remember it's behaviour as a switch statement, so order matters.

Prod error

Here's where the Ruby on Rails comes in. Let's say we're using Ruby on Rails as a server to manage the application and backend. According to the react docs even if we use React Router when a page such as /path is requested the server will look in the build file, in that folder, such is the case for a static site. What we need to do is tell our server that when a resource or page is not found, then render index.html, which will essentially let react router do the routing from there. This was a nightmare to find for a Rails Server, but conveniently addressed in the deployment section of the react-create app docs

routes.rb

Rails.application.routes.draw do  
   ...
   get '*path', to: "application#fallback_index_html", constraints: ->(request) do    !request.xhr? && request.format.html? 
   end
end

A word about SEO optimization
One of the biggest reasons for creating multiple pages is better indexing from search engines. One problem still remains, in that many crawling bots can't execute Javascript properly, not to mention puts a lot of strain on the browser. That means the page takes longer to load, you get indexed as a 'slow page' by pagespeed insights, and you're less likely to enjoy that juicy free traffic. One thing to implement is called SSR or server side rendering, where we do all the heavy lifting on the server and show the user complied HTML files. This dramatically increases load time and SEO discoverability, and is done through frameworks like Gatsby,js. However, React has libraries to implement this functionality as well so you don't have to tango with another dreaded learning curve. This is the topic of another post...
The approach outlined in this post involves the creation of multiple different URL's, showing google your site is readily publishing new content in the case of a blog, and let's it better index your site in search by using the URL in addition to the meta description and title.
